# ZK Wordle-Duel — feasibility spike

A self-contained Noir + Barretenberg spike that validates the hard unknowns of a
**fully trustless, chain-only mutual-duel** Wordle PvP before committing to the
full design. See the plan in `~/.claude/plans/` and `async_pvp_wordle_plan.md`
for the broader game context.

**What this proves (one direction, one guess):** that a 5-tile Wordle `feedback`
is the correct scoring of a public `guess` against a _private_ `secret` word,
that the secret opens a public Poseidon `commitment`, and that the secret is a
real answer word (Poseidon Merkle membership against a public `dictionary_root`).
Turn alternation, escrow, and the second player's direction are deferred.

## Encoding conventions (keep in lock-step everywhere)

- Letters `a..z` → `0..25`.
- Tiles: `absent=0, present=1, correct=2`.
- Feedback packed base-4, least-significant tile first: `Σ tile_i · 4^i`.
- Scoring is a bit-exact port of `src/lib/game.server.ts::evaluateGuess`
  (two-pass, left-to-right first-unused-match). Pinned by tests lifted from
  `src/lib/game.test.ts`.
- Dictionary: Poseidon2 Merkle tree, depth 12 (4096 ≥ 2315 answers), zero-padded.
  Leaf = `poseidon2([l0..l4])`; node = `poseidon2([left, right])`. The TS builder
  (`scripts/build-tree.ts`) and the Noir circuit agree because bb.js Poseidon2 is
  byte-identical to the Noir `poseidon` lib (verified against the commitment).
  `dictionary_root` is the analog of the existing pinned `wordListHash`.

## Commands

bb 5.x replaced `--scheme/--oracle_hash` with `-t <verifier_target>`. We use
`-t evm` = Ethereum/Solidity (keccak transcript, ZK).

```bash
cd circuits
nargo test            # scoring + commitment + membership vectors (10 tests)
nargo execute         # solve witness from Prover.toml -> target/wordle_feedback.gz
nargo compile         # -> target/wordle_feedback.json (ACIR)

bb gates -b target/wordle_feedback.json
bb prove -b target/wordle_feedback.json -w target/wordle_feedback.gz -o target/ -t evm --write_vk
bb verify -k target/vk -p target/proof -i target/public_inputs -t evm
bb write_solidity_verifier -k target/vk -o ../contracts/zk/WordleVerifier.sol

# Commit the proof + public inputs as test fixtures (CI has no ZK toolchain, so
# the Foundry test reads these checked-in vectors — keep them matched to the verifier):
cp target/proof target/public_inputs ../test/zk/fixtures/

# Solidity end-to-end (from repo root):
forge test --match-path "test/zk/WordleVerifier.t.sol" --gas-report
forge build --sizes        # HonkVerifier runtime size vs EIP-170 (24576)

# Proving harness (browser/mobile proxy):
bun install                # in circuits/
bun run scripts/build-tree.ts react  # dictionary root + membership path
bun run scripts/prove.ts             # single-thread WASM (mobile proxy)
BB_THREADS=0 bun run scripts/prove.ts  # multi-thread comparison
```

Note on public inputs: `target/public_inputs` holds the **8** circuit public
inputs `[commitment, dictionary_root, guess[0..5], feedback]`. The generated
verifier reports `NUMBER_OF_PUBLIC_INPUTS = 16` because it adds 8 pairing-point
limbs — but those live **inside the proof**, so `verify(proof, publicInputs)`
takes the 8-word array.

## Metrics (measured — full circuit: scoring + Poseidon2 commitment + Merkle membership)

Numbers in parentheses are the pre-membership baseline (scoring + commitment only).

| Metric                        | Value                                     | "Good" ballpark             | Verdict     |
| ----------------------------- | ----------------------------------------- | --------------------------- | ----------- |
| Gate count (`circuit_size`)   | **7,297** (was 631)                       | low-thousands–low-tens-of-k | ✅ under    |
| Proof size                    | **7,616 B** (was 6,464)                   | ~few KB                     | ✅          |
| Proving time (WASM, 1-thread) | **~380 ms** (was 787 ms)                  | ≲ ~10 s (mobile proxy)      | ✅          |
| Verify gas (Gnosis EVM)       | **~2.27 M** median (was 2.08 M)           | —                           | ⚠️ see note |
| Verifier deploy gas           | ~3.96 M (one-time)                        | —                           | ✅          |
| Verifier runtime bytecode     | **17,829 B** (was 17,827; 6,747 B margin) | **< 24576 (EIP-170)**       | ✅ PASS     |

> **Membership is nearly free on the binding constraints.** Adding the depth-12
> Poseidon Merkle proof took gate count 631 → 7,297, but the verifier **bytecode
> is fixed by the proof system, not circuit size** (17,827 → 17,829 B), and verify
> gas rose only ~185 k. Proving stays well under 1 s single-threaded.

> **Verify gas note:** ~2.27 M is the **ZK** UltraHonk verifier (the ZK variant is
> required to keep the secret word private). That's higher than a non-ZK UltraHonk
> (~600 k), but at Gnosis gas prices (~1–2 gwei, xDAI) it costs a fraction of a
> cent per verification, so gas is **not** the binding constraint.

## Go / no-go — verdict: **GO** ✅

All gates pass for a chain-only mutual-duel ZK PvP on Gnosis with Noir + bb,
**including dictionary membership** in the proven statement:

- **Verifier bytecode 17,829 B < 24,576** (EIP-170) — the hard gate clears with
  6.7 KB to spare, even with the optimizer at 200 runs, and membership barely
  moved it (the verifier size is fixed by the proof system, not circuit size).
- **Verify gas ~2.27 M** — sub-cent on Gnosis; acceptable (it is NOT ~600 k, but
  that figure was for non-ZK; ZK privacy is worth it and Gnosis is cheap).
- **Mobile-proxy proving ~380 ms** single-thread desktop WASM — comfortably under
  10 s. Even a 5–10× real-device penalty (~2–4 s) stays acceptable for a
  per-guess proof.
- **Gate count 7,297** — still low-thousands, with headroom to spare.

**Remaining real-world validation before full build:** (1) prove on actual
iOS Safari / Android WebView (the true worst case vs Node WASM); (2) design the
full duel state machine, on-chain match/escrow contracts, and per-guess proof
submission flow. If a future target chain has expensive gas, or per-duel proof
volume balloons, revisit recursive aggregation. Neither threatens core feasibility.
